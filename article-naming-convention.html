A practical field guide to implementing the ISO 19650 container naming structure — field by field — across models, drawings, documents, and CDE submissions on a live project.

ISO 19650-2:2018 BS EN ISO 19650 2026 Edition
Definition
The ISO 19650 naming convention is a structured string of defined fields that uniquely identifies every information container on a project — from BIM models and drawings to documents and reports. It enables automated CDE processing, consistent cross-discipline referencing, and full audit traceability across the project lifecycle without ambiguity.
Why?
Uniqueness
Every file on a project has a name that uniquely identifies it — no two containers can share the same name, eliminating version confusion and duplication.
How?
Structured Fields
8 defined fields separated by hyphens (-) build the complete naming string — each field carries specific project metadata in a fixed position.
Where?
Across All Files
Applied to every information container — BIM models, drawings, documents, reports, registers — consistently across all disciplines and Appointed Parties.
When?
From Day One
The naming convention must be defined in the BEP before any files are created and enforced from mobilisation — retrofitting is costly and error-prone.

Fields Explained
All 8 Fields — Values, Rules & Examples
Field Name Max Length Rules Common Values Example
Field 1 Project 6 chars Alphanumeric, agreed at project outset, consistent across all parties Project code or abbreviation defined in BEP XYZ / SAA01
Field 2 Originator 6 chars Identifies the Appointed Party that authored the container. Unique per party. ARC, STR, MEP, CIV, FIR, ELE, PLB ARC / MEPCO
Field 3 Volume / System 2 chars Defines the spatial volume, building, or system the container covers. ZZ = whole project. ZZ, B1, B2, T1, EX, UG ZZ / B1
Field 4 Level / Location 2 chars Defines the floor level or zone. 00 = all levels / not applicable. 00, B1, G0, L1, L2, RF 00 / L3
Field 5 Type 2 chars Defines the type of information container. Must use standard codes from ISO 19650 or BEP-defined list. M3 (3D Model), DR (Drawing), SP (Specification), RP (Report), SC (Schedule) M3 / DR
Field 6 Role 1 char Single character identifying the discipline role of the originator. A (Arch), S (Struct), M (Mech), E (Elec), P (Plumb), C (Civil), X (Multi) A / M
Field 7 Number 4 chars Sequential number unique within the project-originator-type combination. Zero-padded. 0001, 0002 ... 9999 0001 / 0025
Field 8 Status & Revision 6 chars Suitability code (S0–S4) followed by revision number (P01, C01 etc.). P = preliminary, C = contract. S0-P01, S2-P03, S3-C01, S4-C02 S4-C01

Common Errors
5 Naming Mistakes to Avoid
1
Inconsistent originator codes across parties
If one party uses ARC and another uses ARCH for the same discipline, automated CDE processing fails. All codes must be standardised in the BEP before mobilisation.
2
Free-text descriptions in the name string
Adding words like "FINAL", "REVISED", or "CLIENT" in the filename breaks the structured format. All context must be captured in the defined fields — never in free text.
3
Reusing container numbers across revisions
The sequential number identifies the container — not the revision. The revision field handles version changes. Reusing numbers creates duplicate references and audit failures.
4
Wrong status code on CDE submission
Submitting an S0 work-in-progress file with an S4 status code bypasses the review cycle. CDE workflows must enforce status validation on upload — not rely on author self-reporting.
5
Applying naming retrospectively
Attempting to rename hundreds of files mid-project breaks CDE links, version histories, and cross-references. The naming convention must be in place before the first file is created.
