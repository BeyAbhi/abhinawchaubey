A practical guide to building and governing a Clash Detection Matrix — calibrated to Middle East project stages (SD, DD, CD, IFC) — defining which disciplines clash against each other, at what tolerance per stage, and how issues are prioritised and resolved. References: BIMForum LOD Spec 2021 · ISO 19650-2 · RIBA POW 2020 · AIA G202 · CIBSE Guide A.

SD · DD · CD · IFC BIMForum · ISO 19650 2026 Edition
Definition
A Clash Detection Matrix is a structured coordination schedule that defines which discipline model pairs are tested against each other, the clash type and tolerance applied at each project stage, and the priority and ownership of each clash test — ensuring systematic, stage-appropriate coordination across all federated models.
What?
Discipline Pairs
Defines which model pairs are tested — Arch vs MEP, Struct vs MEP, MEP vs MEP — no gaps, no missed interfaces.
Type?
Clash Classification
Hard, soft, clearance, or workflow — each clash type has different severity and resolution requirements.
How?
Tolerance per Stage
Tolerances tighten as the project progresses — what is acceptable at Stage 2 is a critical failure at Stage 4.
Who?
Ownership & Priority
Every clash test is assigned an owner, a priority level, and a resolution deadline tied to the programme.
Classification
6 Types of Clashes Defined
🔴
Hard Clash
Two elements physically occupy the same space. Example: a structural beam running through an HVAC duct. Must be resolved before fabrication.
🟡
Soft Clash
Elements are within a defined clearance zone but do not touch. Example: a pipe within 50mm of a structural element — violates maintenance access.
🔵
Workflow Clash
A sequencing or scheduling conflict. Example: two trades occupying the same zone in the same construction week — a programme-level issue.
🟠
Clearance Clash
Elements violate a minimum required clearance for operation, maintenance, or access. Commonly applied to MEP plant, valves, and panels.
⚫
Duplicate Clash
Two identical elements occupy the same location — usually a data or modelling error. Indicates poor model discipline or CDE workflow issues.
🟢
Tolerance Clash
An element falls outside agreed dimensional tolerance. Critical at LOD 400 where shop drawing accuracy is required for fabrication.

Best Practices
5 Rules for Effective Clash Management
1
Filter before you share
Always remove false positives — grids, rooms, linked objects — before distributing results. Cluttered reports destroy team confidence in the process.
2
Stage-appropriate tolerances only
Applying LOD 400 tolerances at Stage 2 generates thousands of irrelevant clashes. Match tolerance to what the models can actually deliver at that stage.
3
Assign before the meeting ends
Every critical and high clash must have a named owner and resolution date before the coordination meeting closes. Unassigned clashes are never resolved.
4
Track trends, not just counts
A rising clash count week-on-week signals a modelling or workflow problem. Use your issue tracker data to identify which discipline pairs are not resolving.
5
Zero hard clashes before fabrication
No model should be issued for fabrication with open hard clashes. Make this a formal stage gate — not an advisory — enforced through the CDE approval workflow.
