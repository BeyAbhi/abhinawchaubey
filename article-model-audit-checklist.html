Model Audit
Checklist

A practical guide to conducting formal BIM model audits at project stage gates — covering what to check, how to score findings, who is responsible, and what constitutes a pass or fail before information exchange.

ISO 19650-2 QA / QC 2026 Edition
Definition
A Model Audit Checklist is a structured quality assurance instrument used by the BIM Lead or Information Manager to formally assess model compliance prior to each stage gate information exchange — verifying that every information container meets the BEP standards, achieves the required LOD, and is free of critical coordination and data failures before acceptance into the CDE.
What?
Compliance Check
Verifies that every model meets BEP commitments — naming, LOD, parameters, geometry, file structure, and CDE metadata — before it is accepted for information exchange.
When?
Stage Gate Trigger
Conducted at every major stage gate — SD, DD, CD, and IFC — before the model is elevated from S3 (For Review) to S4 (Approved for Construction) in the CDE.
Who?
BIM Lead / Info Mgr
The audit is conducted by the BIM Lead or Information Manager — independent of the authoring team — to ensure objectivity and eliminate self-certification bias.
Result?
Pass / Fail Decision
The audit produces a formal Pass, Conditional Pass, or Fail outcome — with a scored finding log that drives mandatory resolution before the model can proceed.
Scope
6 Categories Every Model Audit Must Cover
📁
File & Naming
Container name compliance with ISO 19650 structure, file format, revision control, and CDE upload metadata.
📐
Geometry & LOD
Model geometry accuracy, element completeness, LOD achievement against BEP commitments for the current stage.
🏷️
Data & Parameters
Parameter population, property set completeness, LOI achievement, and data consistency across all elements.
🔗
Coordination
Open clash count, unresolved critical issues, model origin and shared coordinates, and federation alignment.
⚙️
Model Health
File size, purge status, in-place families, workset structure, unused elements, and software version compliance.
🔒
CDE Compliance
Suitability code, revision sequence, approval workflow completion, and access permissions within the CDE.

Finding Classification
Audit Finding Severity Scale
Critical
Stop — Do Not Issue
Model cannot proceed to CDE or stage gate. Must be resolved and re-audited before any further action.
Major
Conditional Pass
Model may be issued S3 for review only. Resolution required before S4 approval. Named owner and date mandatory.
Minor
Pass with Action
Model may proceed. Resolution required within agreed timeframe. Logged in audit register for follow-up.
Observation
Information Only
No action required. Noted for awareness and continuous improvement. Does not affect model status.

Common Errors
5 Audit Mistakes to Avoid
1
Author auditing their own model
Self-certification is not an audit. The auditor must be independent of the authoring team — the BIM Lead or Information Manager, never the model author reviewing their own work.
2
No severity classification on findings
An audit report that lists findings without Critical / Major / Minor classification provides no basis for the pass/fail decision. Every finding must be scored before the report is issued.
3
Auditing at S4 — not before
Auditing after S4 approval is too late — the model is already accepted. The audit must occur at S3 (For Review) stage, before approval is granted. Not after.
4
Findings issued with no resolution deadline
Every finding must carry a named owner and a resolution date. An audit report without deadlines generates awareness without accountability — findings are never closed.
5
Re-audit skipped after critical finding resolution
When a critical finding is resolved, the model must be re-audited before S4 approval. Accepting the author's word that it is fixed — without re-checking — defeats the purpose of the audit process entirely.
